# v0.5 plan

> ## Status: shipped
> v0.5.0 shipped 2026-05-04 (same day as v0.4.0). Tag [`v0.5.0`](https://github.com/arijit-gogoi/jolly-http/releases/tag/v0.5.0). This file is the historical plan, kept as a release artifact (linked from the GitHub release notes). For the current contract see [SPEC.md](SPEC.md).

**Theme:** consumer-feedback release, round 2. Vaaktaa shipped against v0.4 and came back with three concrete asks plus one explicit retraction. v0.5 absorbs all four. Same operating model as the v0.4 plan: the only consumer is the spec.

The frozen contract gains two new optional named exports (`before`, `after`), one new helper (`log.event`), and an additive third arg on `default`. One existing default flips: `RequestInit.redirect` defaults per-method.

## Why this shape

Three real pains that v0.4 didn't solve:

1. **`prologue`/`epilogue` are process-level. Cleanup state in vaaktaa is per-iteration** — test users created in `default`, deleted at the end of the same iteration. v0.4's epilogue runs once; the workflow had to fall back to `try/finally` boilerplate. ~150–200 lines across the suite. **`before`/`after` per-iteration hooks** with a context object passed `before → default → after` eliminate the boilerplate and remove the "what if I forget the `if (email !== null)` guard" foot-gun.
2. **Default `redirect: "follow"` hides POST→303 entirely.** Vaaktaa is htmx/Rails-shaped — every login, signup, become-creator, autosave returns 303. Every redirect-emitting POST in the suite passes `redirect: "manual"` (~30 sites). v0.4 documented the footgun in Troubleshooting; documenting isn't fixing. **Per-method redirect default** flips the calculus.
3. **Mid-test trace points have no API.** The v0.4 README pointed at `process.stderr.write(JSON.stringify(...))` as a workaround. **`log.event(name, data?)` writes to the same NDJSON sink as request samples**, with the same envelope. Same downstream tooling reads both.
4. **Retraction:** the consumer's earlier `expected()` ask (from v0.3.1 review) is withdrawn — v0.4's automatic AssertionError context made it a non-issue, and a chai-style chain API would compromise the "real JS, no DSL" identity.

Outcome: vaaktaa drops ~150–200 lines of `try/finally`, ~30 lines of `redirect: "manual"`, gains structured trace points without changing tooling.

## Tracks (priority order)

### Track A — per-method redirect defaults

`src/request.ts`: lookup table `REDIRECT_DEFAULT_BY_METHOD` mapping `GET`/`HEAD`/`OPTIONS → "follow"` and `POST`/`PUT`/`PATCH`/`DELETE → "manual"`. Single line change at the `fetch` call site picks the per-method default if the user didn't override.

**Breaking:** v0.4 workflows that POST and silently followed redirects must add `redirect: "follow"` per call. Pre-1.0 minor bump (0.5.0).

**Tests** (`test/request.test.ts`): 6 tests against a local 303 server — GET follows, POST/PUT/PATCH/DELETE stop at 303, HEAD/OPTIONS follow, explicit overrides work both ways.

### Track B — `log.event` + `SampleEvent`

`src/types.ts`: new `SampleEvent` interface. `Sample = SampleSuccess | SampleError | SampleEvent`. `SamplePhase` extends to `"before"` / `"after"` (Track C will use these but it's cheap to add now).

`src/runtime.ts`: `log = { event(name, data?) { ... } }` calls `currentContext().sink.write({ ok: true, t, vu, iteration, phase?, event, data?, ts })`.

`src/load.ts` `Stats.push`: defensive `if ("event" in sample) return` guard — events bypass HTTP-outcome stats. Today nothing routes events through onOutcome, but the guard documents the invariant.

`src/index.ts`: export `log`, `SampleEvent`.

**Tests** (`test/runtime.test.ts`): 7 tests — basic shape, optional `data`, omits `data` when undefined, carries `phase` from context, omits `phase` on iteration ctx, throws outside workflow, vu/iteration come from context.

### Track C — per-iteration `before` / `after` hooks

**Types** (`src/types.ts`):
- `BeforeFn = (vu, signal) => Promise<unknown> | unknown`
- `AfterFn = (vu, signal, ctx) => Promise<void> | void`
- `WorkflowFn` gains optional third arg `ctx?: unknown` (additive)
- `SamplePhase += "before" | "after"`

**Loader** (`src/run.ts`):
- `LoadedWorkflow` gains `before?`, `after?`.
- `loadWorkflowModule` validates them as functions if present.
- Arity warning bumped from `> 2` to `> 3` (third arg `ctx` is now legitimate).

**Single-run wiring** (`src/run.ts`): inside the existing `scope({signal: opts.signal}, ...)`, the iteration phase replaces its single `withRuntime` call with a nested per-iteration `scope({signal: s.signal}, async iterScope => ...)`. Inside that iteration scope, `after` registers as `iterScope.resource(true, async () => ...)` BEFORE `before` runs. LIFO release means `after` fires whether `before` succeeded, threw, or got aborted. `userCtx` is captured by closure so `after` sees the live value `before` returned, even when `default` throws mid-iteration.

**Load-mode wiring** (`src/load.ts`): `oneIteration` opens a nested scope per iteration with the same `after`-as-resource pattern. `VuCtx` gains `before?`, `after?`. Hooks reference threaded from `mod.before` / `mod.after` at the `pool.spawn(() => runVu(...))` site.

**Per-iteration RuntimeContext rules:**
- `phase: "before"` / `phase: "after"` set on the per-phase context.
- Cookie jar SHARED across phases of the same iteration.
- `lastResponse` ISOLATED per phase (separate scratch slot).
- Iteration index is the REAL iteration index — phase tag is the discriminator.

**Stderr noise:** `after` may legitimately throw structural cancellation when scope deadline fires mid-iteration. Suppress via `isStructuralCancellation(err) || ctx.signal.aborted` check before logging. (Discovered while running the load-mode tests.)

**Tests:**
- `test/run.test.ts`: 9 tests covering happy path, before-throws-after-still-runs (epilogue-shaped guarantee at iteration scale), **discriminating test** (default throws and after reads ctx that `before` returned — proves the resource captured the live ref), 3-arg arity does not warn, 4-arg DOES warn, cookie jar shared across phases, NDJSON phase tags, rejects non-function before/after.
- `test/load.test.ts`: 2 tests covering per-VU per-iteration counts and after-runs-on-default-throw at load.

## API surface diff

**New exports** (from `jolly-http`):
- `log` (namespaced, `.event(name, data?)`)
- types: `BeforeFn`, `AfterFn`, `SampleEvent`

**Changed types:**
- `Sample = SampleSuccess | SampleError | SampleEvent`
- `SamplePhase += "before" | "after"`
- `WorkflowFn` third arg `ctx?: unknown` (additive — existing 2-arg signatures still match)
- `LoadedWorkflow`/`WorkflowHooks` gain `before?`, `after?`

**Changed defaults (BREAKING):**
- `RequestInit.redirect` per-method (POST/PUT/PATCH/DELETE → `"manual"`).

**Removed/deprecated:** none.

## Out of scope (reaffirmed + new)

- `request.use(middleware)` — wrapper modules in userland are the pattern.
- `expect()` parallel API — `import { expect } from "vitest"` works.
- **`expected()` (proposed by consumer in v0.3.1, withdrawn in v0.5).** The original case was "syntactic sugar for negative assertions." v0.4's automatic AssertionError context made the case obsolete: `assert(login.status === 303, "...")` already auto-attaches request URL/status/headers/full body on failure, narrows types via `asserts cond`, and has zero new vocabulary. A chai-style chain would be feature creep against the "real JS" identity.
- `toMatchSnapshot()` — no demonstrated user pain.
- `pino`-style logger — `log.event(name, data?)` covers structured trace points.
- CSRF helpers — too domain-specific.
- JSON-schema response assertion — `import { z } from "zod"` works in `.mjs`/`.ts`.

PSL for cookies / loose HAR matching / mode-based env chains continue to defer (now to v0.6+).

## Order of work

1. Track A (smallest blast radius; 1 line + table) → flush out test pain first.
2. Track B (purely additive types + one export) → so README examples can use it.
3. Track C (largest churn; load.ts oneIteration redesign) → builds on stable A and B.

## Per-iteration scope: nested scope, not try/finally

Decision: nested `scope({ signal }, async iterScope => ...)` per iteration; `after` registered via `iterScope.resource(true, async () => ...)` BEFORE `before` runs.

Why:
- Symmetry with v0.4 epilogue registration (`run.ts:223`) — same pattern, same mental model.
- "After runs when before threw" guarantee falls out of LIFO resource disposal — no manual try/finally bookkeeping.
- Performance: scope creation is allocation-cheap; iteration is gated by network I/O. `runLoad` already nests scopes (`scope({ limit, signal }, async pool => ...)`).

Trade-off accepted: each iteration spawns one extra scope object. Negligible vs `fetch` overhead.

## DoD for the release

- All three tracks meet their per-track DoD.
- 247+ vitest tests passing (was 223 in v0.4, +24 in v0.5).
- Three-mode smoke (`test/fixtures/smoke/smoke.mjs`) extended to cover `before`/`after`, `log.event`, and the per-method redirect default — passes against the just-built `dist/cli.js`.

- Grep for stale `[v0.5]` annotations before tag (per `feedback_post_ship_audit_pattern`).

## What this release deliberately is not

- Not a flagship-driven moat-deepening cosmetic push. The flagship *is* fixing what bit a real user.
- Not where snapshot testing, middleware, `expect()`, `expected()`, or `pino` lands.
- Not a v1.0 candidate. Pre-1.0 breaking changes remain the model.
