# v0.4 plan

> ## Status: shipped
> v0.4.0 shipped 2026-05-04. Tag [`v0.4.0`](https://github.com/arijit-gogoi/jolly-http/releases/tag/v0.4.0). This file is the historical plan, kept as a release artifact (linked from the GitHub release notes). For the current contract see [SPEC.md](SPEC.md).

**Theme:** consumer-feedback release. Vaaktaa is the only production consumer. Their pain points are the spec.

The frozen contract in [SPEC.md §3](SPEC.md#3-workflow-file-format-frozen) gains two new optional named exports (`prologue`, `epilogue`). Existing workflows are unchanged. One existing default flips: `--cookies <dir>` no longer auto-loads prior-session jars.

## Why this shape

The first plan I wrote for v0.4 (PSL + HAR loose-match + env-mode chains + TS workflows + `.d.ts` polish) was internally consistent but priority-wrong. None of those came up when a real user shipped against jolly-http on a real project. What did come up:

- The cookie default bit them in CI (`reference_jolly_http_quirks.md` exists in their memory because of this).
- `fetch failed` told them nothing, costing hours of "is it DNS, abort, or downstream timeout?"
- Assertion failures gave them a one-line message with no request/response context.
- `try/finally { await cleanupUser(email) }` was hand-written in every suite.
- POST→303 was invisible by default (server-rendered apps are the target audience, this should be obvious).

Those are v0.4. Everything else slips.

## Tracks (priority order)

### Track 1 — Cookie default flip [breaking]

**Problem:** `--cookies <dir>` currently loads any prior-session jar from disk on startup ("session continuity by design"). When a test suite fails halfway, the next run inherits a logged-in state and assertions silently misbehave. Vaaktaa wrote a `clean:cookies` npm script to work around it.

**Decision:** flip the default. `--cookies <dir>` becomes write-only audit trail — every run starts with a fresh in-memory jar; the dir gets written on exit. Add `--cookies-resume <dir>` for the rare case the old behavior is wanted (true `httpie --session=name` semantics).

**Surface:**
- `--cookies <dir>` — write-on-exit. Saves to `<dir>/vu-N.json`. Does not load.
- `--cookies-resume <dir>` — load-from-disk-then-save. The old v0.3 behavior of `--cookies`.
- No `--cookies` at all — in-memory only, discarded on exit. (Unchanged.)

**DoD:**
- Test: `--cookies ./jar` twice in a row; the second run does NOT inherit cookies from the first.
- Test: `--cookies-resume ./jar` twice in a row; the second run DOES inherit cookies (old behavior preserved under the new flag).
- README cookies section rewritten — "fresh-each-run is the default; opt in to resume."
- CHANGELOG entry calls out the breaking change explicitly.

**Pre-1.0 minor bump (0.4.0). Breaking but allowed pre-1.0.**

### Track 2 — Assertion failure context

**Problem:** when `assert(profile.status === 200, "C5: expected 200, got 404")` fails, the user sees only the message. Nothing about which request, no response body, no headers. Debugging requires re-running with `console.log` everywhere.

**Decision:** track last-seen `Response` per VU in the runtime context. `AssertionError` formatting auto-appends:

```
AssertionError: C5: expected 200, got 404
  → last request: GET https://api.example.com/api/me 404
  → response headers: { "content-type": "application/json", ... }
  → response body: {"error":"not authenticated","trace_id":"abc123"}
```

**No body truncation.** Print the full body. Debugging needs everything; the user is reading the error in a terminal, they can scroll.

**Surface:**
- Internal: `RuntimeContext` gains `lastResponse?: { method, url, status, headers, bodyText }`.
- `performRequest` updates it after every successful request (and on non-2xx — the body is what they need most).
- `AssertionError.toString()` (or `.message` formatter) reads from runtime context if set.
- No public API change. No new exports.

**DoD:**
- Test: `assert(false)` after a `request.GET` includes the request URL, status, headers, and full body in the error message.
- Test: `assert(false)` with no prior request shows just the original assertion message (no "last request" suffix).
- README assertion section gains an example showing the formatted error.

### Track 3 — `fetch failed` cause chain

**Problem:** undici throws bare `TypeError: fetch failed` for everything: DNS failure, connection refused, abort, downstream timeout. The user has no idea where to look.

**Decision:** in `performRequest`, when the `fetch` call throws, walk `err.cause` and surface a structured error. Common causes get readable names:

- `ECONNREFUSED` / `ENOTFOUND` / `ETIMEDOUT` / `ECONNRESET` — network errors with the system errno
- `AbortError` — already named, preserve
- `UND_ERR_HEADERS_TIMEOUT` / `UND_ERR_BODY_TIMEOUT` — undici-specific, surface as-is
- Anything else — preserve `err.cause.message` in the wrapped error message

**Surface:**
- `Sample` (NDJSON) `error` field gets the structured name; `message` field gets the human description.
- Thrown errors from `request.GET/POST/...` carry `.cause` if undici provided one.
- No new public API.

**DoD:**
- Test: request to a closed port → `Sample.error === "ECONNREFUSED"`.
- Test: request to nonexistent domain → `Sample.error === "ENOTFOUND"`.
- Test: aborted request → `Sample.error === "AbortError"` (unchanged).
- README troubleshooting section (Track 5) lists these and their causes.

### Track 4 — `prologue` / `epilogue` hooks

**Problem:** every real suite ends with `try/finally { await cleanupUser(email) }`. Setup code (creating a test user, getting a CSRF token, setting up DB fixtures) lives at the top of the workflow function and runs once per VU per iteration in load mode — wasteful when it should run once per process.

**Decision:** two new optional named exports. Frozen API addition.

```js
// flow.mjs
import { request, env } from "jolly-http"

export async function prologue(env, signal) {
  // runs ONCE per process, before any VU/iteration
  // env is Readonly<Record<string,string>> — same shape as vu.env
  // signal aborts on Ctrl-C / fatal
  // throw to abort the run before workflow body runs
  // return value is discarded (or → consider exposing to default? open question)
}

export default async function (vu, signal) {
  // unchanged — runs per VU per iteration
}

export async function epilogue(env, signal) {
  // runs ONCE per process, AFTER all VUs/iterations finish OR on abort/Ctrl-C
  // wrapped in scope resource semantics so it ALWAYS runs (LIFO with sink, jar)
  // throw to mark the run as failed even if all iterations passed
}
```

**Semantics:**
- `prologue` runs in the parent scope before any VU starts. Errors abort the run with exit 1 before any iteration runs.
- `epilogue` runs in the parent scope as a `s.resource` cleanup — guaranteed to run on success, failure, or Ctrl-C. LIFO with sample sink and cookie jar.
- Both receive `(env, signal)` — no `vu` parameter. They're not per-VU.
- Single-run mode (one iteration): prologue → default → epilogue, sequential.
- Load mode: prologue → spawn N VUs each running default in a loop until duration → epilogue.
- Watch mode: prologue/epilogue run on every reload, scoped to that run.

**Open questions (resolve during implementation, not now):**
- Does `prologue` return value get exposed to the default function? Probably no — keeps the surface small. Workflow authors can stash setup state in module-level `let` bindings (real JS).
- What if a user defines `prologue` but no default? Probably treat as one-shot (run prologue, run epilogue, exit). Or error. Decide during impl.

**Surface:**
- Two new optional named exports on the workflow file.
- No new CLI flags. No new runtime imports.

**DoD:**
- Test: prologue runs once before any iteration; epilogue runs once after.
- Test: epilogue runs even when default throws (resource discipline).
- Test: epilogue runs on SIGINT.
- Test: throwing from prologue exits with code 1 and skips default + epilogue (decision: epilogue does NOT run if prologue threw — there's nothing to clean up).
- README new section: "Setup and teardown."

**One advisor call before I commit the exact signature** — frozen API additions are permanent.

### Track 5 — Troubleshooting README section

**Problem:** when things go wrong, there is nowhere to look.

**Decision:** new top-level README section "Troubleshooting." One paragraph each:

- **`fetch failed`** — cross-link Track 3. Show the wrapped error format.
- **POST returned no error but assertion fails** — explain default `redirect: "follow"`. Show `redirect: "manual"` recipe for POST→303 patterns. Server-rendered apps (htmx, Rails, Phoenix) need this.
- **`"runtime used outside a workflow"`** — what counts as a workflow file (cross-link Track 6).
- **Missing env var** — `--require-env` pattern.
- **Cookies surviving across runs** — only happens with `--cookies-resume`; the default `--cookies` is fresh-each-run (cross-link Track 1).
- **Structured logging in load mode** — recommended pattern: `process.stderr.write(JSON.stringify({...}) + "\n")` for trace points that survive concurrent VUs without garbling.

**DoD:**
- Section exists, every listed item present, each cross-link resolves.

---

### Track 6 — "What counts as a workflow"

**Problem:** importing `request` into a non-workflow `.mjs` (a helper module called at import time) throws "jolly-http runtime used outside a workflow" with no explanation. Mechanism (AsyncLocalStorage context-bound runtime) is undocumented.

**Decision:**
- Improve the error message:
  ```
  jolly-http: request/assert/env/sleep can only be used from inside a workflow function.
  Helper modules are fine — but the helper has to be CALLED from a workflow function,
  not run at import time. Move the call inside your default export.
  ```
- README new subsection under workflow API: "What counts as a workflow."

**DoD:**
- Test: import-time call to `request.GET` produces the new error message.
- README section exists, explains "called from inside default/prologue/epilogue is fine; module-top-level is not."

## Demoted / deferred to v0.5+

- **PSL for cookies** — cookie default flip eliminates most cross-tenant-leak risk in the only known consumer. Real PSL handling can wait.
- **HAR loose-match** — no consumer feedback hit this.
- **Mode env-file chains** — no consumer feedback hit this.
- **DB/Redis/S3 companion packages** (`jolly-http-pg`) — vaaktaa BYO `pg` import. Worth shipping eventually, not yet — need 2+ suites to know the API shape.
- **Default `redirect: "manual"` for POST** — too breaking. Track 5 docs cover it.
- **`pino`-style logging** — `process.stderr.write(JSON.stringify(...))` is documented in Track 5.
- **CSRF helpers** — too domain-specific; user can write a 5-line helper.
- **JSON-schema response assertion** — `import { z } from "zod"` works in `.mjs`.

## Out of scope (deliberate, with reasons)

- **`request.use(middleware)`** — wrapper modules in userland are the pattern. Permanent surface (ordering, async, ctx shape, retry semantics) we'd regret.
- **`expect()` parallel API** — `import { expect } from "vitest"` works. Don't duplicate the frozen `assert` API.
- **`toMatchSnapshot()`** — no demonstrated user pain. The cookie + assertion fixes might surface real pain that justifies this in v0.5; revisit then.

## Order of work

1. Track 2 (assertion context) — small, reversible, immediate DX win
2. Track 3 (cause chain) — small, no API change
3. Track 6 (runtime-outside-workflow error message) — tiny, paired naturally with Track 2
4. Track 1 (cookie flip) — breaking; do it once, deliberately
5. Track 4 (prologue / epilogue) — frozen API addition; advisor call first
6. Track 5 (README troubleshooting)
7. Track 7 (TS + .d.ts)

## DoD for the release

- All seven tracks meet their per-track DoD.
- CHANGELOG entry: "Breaking — `--cookies <dir>` no longer auto-loads prior-session jars. Use `--cookies-resume <dir>` for the previous behavior."
- Three-mode smoke test (per `reference_published_package_smoke_pattern`) passes against the published v0.4.0.
- Vaaktaa upgrades cleanly:
  - Their `clean:cookies` npm script becomes redundant.
  - Their assertion error messages now include request context.
  - Their `try/finally` boilerplate becomes a single `epilogue` export.
  - `fetch failed` outputs are diagnosable.
- Grep for stale `[v0.4]` annotations before tag (per `feedback_post_ship_audit_pattern`).

## What this release deliberately is not

- Not flagship-driven. The flagship *is* fixing what bit a real user.
- Not where snapshot testing, middleware, or `expect()` lands.
- Not a moat-deepening cosmetic push. The moat is real users shipping real things; v0.4 makes that less painful.
